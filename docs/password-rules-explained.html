1. ה-REGEX המלא — אנטומיה שלמה

להלן ה-REGEX המלא לאימות סיסמא חדשה. הכלל הנדון בדף זה הוא כלל 5 — ה-Negative Lookahead שאוסר תווים ספציפיים:

^(?=.*[A-Z])(?=.*[a-z])(?=.*\d)(?=.*[!@#$%&*,./;<>|{}\[\]"'\\])(?!.*[ ~`':^()_+\-=?]).{10,50}$
^    $ עוגני תחילה וסוף — הרגקס חייב לכסות את כל המחרוזת, לא רק חלק ממנה
(?=.*[A-Z]) Lookahead חיובי — חובה לפחות אות גדולה אחת
(?=.*[a-z]) Lookahead חיובי — חובה לפחות אות קטנה אחת
(?=.*\d) Lookahead חיובי — חובה לפחות ספרה אחת (0–9)
(?=.*[!@#…]) Lookahead חיובי — חובה לפחות תו מיוחד מהרשימה המותרת
(?!.*[ ~`':^()_+\-=?]) ⬅ כלל 5 — זה הכלל שמוסבר בדף זה: Negative Lookahead — אסור שיופיע אפילו תו אחד מהרשימה
.{10,50} אורך: מינימום 10 תווים, מקסימום 50 תווים
מה זה Negative Lookahead? (?!...) הוא assertion — הוא בודק את מה שמגיע אחריו מבלי לצרוך תווים. (?!.*X) אומר: "ממיקום הנוכחי, אל תרשה שיופיע X בשום מקום בהמשך המחרוזת". מכיוון שהוא ממוקם אחרי ^, הוא בודק מתחילת המחרוזת עד סופה.

3. איך הדפדפן בודק זאת — ללא UI

מנוע ה-JavaScript בדפדפן (V8 ב-Chrome, SpiderMonkey ב-Firefox) מריץ את ה-REGEX כ-ECMAScript RegExp. ניתן לבדוק זאת ישירות ב-DevTools Console (F12 ← לשונית Console) ללא שום UI, ללא שדות קלט, ללא כפתורים.

כיצד RegExp.test() עובד

  1. מחרוזת הסיסמא עוברת את ה-REGEX מתחילתה
  2. כל lookahead נבדק במקביל (zero-width)
  3. ה-Negative Lookahead בודק: האם קיים תו מהרשימה?
  4. אם כן — false. אם לא — ממשיך לכלל הבא
  5. בסוף: .{10,50} מוודא אורך
  6. מחזיר true רק אם כל התנאים עברו

מה בדיוק (?!.*X) עושה

  • הדפדפן ממקם "מצביע" בתחילת המחרוזת (אחרי ^)
  • .* — קופץ קדימה 0 עד N תווים (greedy)
  • מחפש את X (תו אסור) בכל עמדה
  • אם נמצא: ה-assertion נכשל → כל ה-REGEX מחזיר false
  • אם לא נמצא: ה-assertion עובר → ממשיך
  • זהו backtracking — הדפדפן מנסה כל עמדה
הפרדה חשובה: REGEX vs validation בצד שרת הבדיקה שב-DevTools בודקת את לוגיקת ה-REGEX בלבד כפי שמוגדר בקוד הלקוח. הבדיקה בפועל (לחצן "שמור סיסמא") עשויה לכלול גם validation נוסף בצד השרת. ה-DevTools נותן לנו ביטחון שה-REGEX עצמו נכון — לפני שגולל ב-UI.

6. שאלות נפוצות (FAQ)

למה - (מינוס) אסור אבל / (לוכסן) מותר?

ה-REGEX הוגדר כך על ידי מפתחי המערכת. ייתכן שמינוס אסור כי מערכות מסוימות משתמשות בו כ-delimiter בפורמטים פנימיים (כגון JWT, tokens). הלוכסן לעומת זאת שכיח בסיסמאות ולא יוצר בעיות parsing.

למה _ (underscore) אסור? זה נפוץ בסיסמאות!

החלטה של המערכת — underscore אסור. זה בגדר דרישה, לא טעות. כ-QA, הבדיקה שלנו מוודאת שה-REGEX מיישם דרישה זו, לא שואל אם היא נכונה.

מה ההבדל בין + שאסור לבין * שלא ברשימה האסורה?

* אינו ברשימת האסורים ואינו ברשימת המותרים המפורשת. כלומר, * יכול להופיע בסיסמא תקינה! זאת בניגוד ל-+ שאסור מפורשות. בדוק זאת: checkPassword('Multi*1!pass') — ה-DevTools יאמר ✅ PASS.

אחרי הדבקת הסקריפט ב-DevTools, קיבלתי שגיאה. מה לעשות?

  • ודא שהדבקת את כל הסקריפט (מ-const PASSWORD_REGEX עד השורה האחרונה)
  • אם קיבלת SyntaxError — נסה לפתוח טאב חדש ריק (about:blank) ואז הרץ
  • CSP של דפים מסוימים עלול לחסום — הרץ מ-about:blank תמיד

כיצד להריץ על about:blank?

  1. פתח טאב חדש בדפדפן
  2. בשורת הכתובת הקלד about:blank ולחץ Enter
  3. פתח DevTools: F12 ← Console
  4. הדבק את הסקריפט ולחץ Enter
  5. הפונקציות זמינות: checkPassword('...'), runAllTests(), etc.

    העתק את כל תוכן בלוק הקוד. הדבק ב-Console עם Ctrl+V. המסוף יאפשר הדבקת קוד מרובה שורות.
  3. לחץ Enter
    הסקריפט ירוץ אוטומטית: יציג קודם את כל 13 התווים האסורים, אחר כך טבלת 41 מקרי בדיקה, ואחר כך 4 בדיקות בודדות.
  4. בדוק תו ספציפי
    הקלד ב-Console: checkPassword('Mytest1!pass') — מקבל פירוט של כל 6 כללים. או: checkSingleForbiddenChar('_') לבדיקת תו בודד.
  5. השתמש ב-console.table
    runAllTests() פולט טבלה ב-DevTools שניתן למיין לפי עמודה — לחץ על כותרת העמודה.
  6. בדיקת מחרוזת מה-clipboard
    ניתן גם: checkPassword(await navigator.clipboard.readText()) כדי לבדוק סיסמא שהועתקה — ללא הקלדה ידנית.
אזהרה — CSP (Content Security Policy) דפים מאובטחים עלולים לחסום הרצת קוד ב-Console. אם קיבלת הודעה Refused to evaluate — פתח טאב נפרד עם about:blank והרץ משם. הסקריפט אינו תלוי ב-DOM של העמוד — הוא עובד על כל עמוד.

SCOPE ניהול משתנים Scope, Hoisting & TDZ

Scope = תחום הנראות של משתנה. JS יש שלושה סוגים:

  • Global scope — מחוץ לכל פונקציה, נגיש מכל מקום.
  • Function scope — var חי בתוך הפונקציה בלבד.
  • Block scope — const/let חיים בין { }.
var x = 1; // function-scoped — "מוגבה" לראש הפונקציה let y = 2; // block-scoped — TDZ עד שורת ההצהרה const z = 3; // block-scoped — גם לא ניתן לשינוי // Hoisting: var מוגבה (אבל ערכו undefined עד ההשמה) console.log(a); // undefined — לא שגיאה! var a = 10; // TDZ: let/const מוגבים אבל לא נגישים לפני ההצהרה console.log(b); // ReferenceError: Cannot access 'b' before initialization let b = 10;
אנלוגיהvar כמו שלט שתלית בחדר — כולם רואים אותו גם לפני שכתבת עליו. let/const כמו תיבה נעולה — קיימת אבל לא ניתנת לפתיחה לפני שקיבלת מפתח.
מלכודת נפוצהvar בתוך לולאת for חולק את אותו scope עם כל האיטרציות — תקבלי תוצאות מפתיעות. השתמשי תמיד ב-let בלולאות.
בסקריפט שלנוכל המשתנים הוגדרו עם const ברמת הסקריפט — REGEX_RULES, TEST_CASES, FORBIDDEN_CHARS. זה מונע שינוי מקרי ומעניק block scope ברור.
FUNCTION פונקציות Arrow Function & קשירת this

Arrow function (=>) היא תחביר קצר לפונקציה, אך ההבדל העיקרי הוא אין לה this משלה — היא לוקחת את this מהסביבה שבה הוגדרה (lexical this).

// פונקציה רגילה — this תלוי במי קרא לה function greet() { return `Hello ${this.name}`; } const obj = { name: 'Maya', greet }; obj.greet(); // "Hello Maya" — this = obj // Arrow — this נקבע ב-definition time, לא call time const obj2 = { name: 'Maya', greet: () => `Hello ${this.name}` // this = global/undefined! }; obj2.greet(); // "Hello undefined" // תחביר קצר — ערך יחיד, ללא return מפורש const double = x => x * 2; const add = (a, b) => a + b;
אנלוגיהפונקציה רגילה = שכיר שמדווח למנהל שקרא לו. Arrow = פרילנסר שמדווח תמיד למשרד הבית שלו, לא משנה מי שכר אותו.
מלכודת נפוצהאל תשתמשי ב-Arrow לפונקציות של object methods כשאת צריכה this. גם לא תוכלי להשתמש ב-arguments בתוך arrow — השתמשי ב-rest params (...args).
בסקריפט שלנוTEST_CASES.forEach(tc => { ... }) — arrow מושלמת כאן כי אנחנו לא צריכים this בתוך ה-callback. גם REGEX_RULES.every(r => r.test(pw)).
CLOSURE פונקציות Closure — פונקציה שזוכרת את הסביבה שלה

Closure נוצר כשפונקציה "זוכרת" משתנים מהסקופ שבו היא נוצרה, גם אחרי שאותו סקופ כבר לא ב-call stack.

function makeCounter(start = 0) { let count = start; // חי ב-scope של makeCounter return () => { count++; // הפנייה לאותו count — closure! return count; }; } const c1 = makeCounter(); const c2 = makeCounter(10); c1(); // 1 c1(); // 2 c2(); // 11 — scope נפרד לגמרי!
אנלוגיהתאר לך ארנק שנוצר בבית ספר — הוא עדיין מחזיק כרטיס תלמיד גם אחרי שגמרת ת"ז. הארנק = פונקציה, כרטיס = משתנה מה-scope המקורי.
מלכודת נפוצהvar בלולאה + closure = כולם מצביעים לאותו משתנה. for(var i=0;i<3;i++) setTimeout(()=>console.log(i)) יפלוט 3,3,3. עם let יפלוט 0,1,2.
בסקריפט שלנוcheckPassword יוצרת closure על REGEX_RULES ו-FORBIDDEN_CHARS המוגדרים ב-scope החיצוני — היא ניגשת אליהם בכל קריאה בלי לקבלם כפרמטרים.
PROTOTYPE OOP Prototype Chain — ירושה ב-JavaScript

כל object ב-JS יש לו __proto__ — מצביע ל-object אחר שממנו הוא "יורש" מתודות. זהו prototype chain.

// String.prototype מחזיק את כל מתודות המחרוזת 'hello'.toUpperCase(); // 'hello'.__proto__ === String.prototype // Array.prototype: map, filter, forEach, every… [1,2,3].map(x => x*2); // [1,2,3].__proto__ === Array.prototype // חיפוש: JS עולה בשרשרת עד שמוצא או מגיע ל-null const obj = { a: 1 }; obj.toString(); // לא על obj עצמו — נמצא ב-Object.prototype // Monkey Patching — הוספת מתודה לכל objects מסוג מסוים String.prototype.isPalindrome = function() { return this === [...this].reverse().join(''); }; 'racecar'.isPalindrome(); // true — אבל זה מסוכן בפרודקשן!
אנלוגיהספר מתכונים משפחתי — כשאת מבשלת ומחפשת מתכון, קודם מסתכלת בגרסה שלך. אם אין — בגרסת אמא. אם אין — בגרסת סבתא. זהו prototype chain.
מלכודת נפוצהMonkey Patching על Built-in prototypes (String, Array, Object) עלול לשבור ספריות צד-שלישי שמניחות שהפרוטוטייפ המקורי. הימנעי מזה בקוד production.
בסקריפט שלנוpassword.split('').some(c => FORBIDDEN_CHARS.has(c)) — split ו-some הם מתודות מ-String.prototype ו-Array.prototype. Set.prototype.has — O(1) מהיר מאוד.
REGEXP ביטויים רגולריים RegExp מעמיק — lookahead, flags, groups

RegExp (Regular Expression) = תבנית לחיפוש / בדיקת מחרוזות. /pattern/flags

// Flags חשובים const re1 = /hello/i; // i = case-insensitive const re2 = /\d+/g; // g = global — מוצא כל ההתאמות const re3 = /^line$/m; // m = multiline — ^ ו-$ לכל שורה // Character classes /\d/ // ≡ [0-9] ספרה /\w/ // ≡ [a-zA-Z0-9_] מילה /\s/ // רווח/טאב/שורה חדשה /\D/ // לא-ספרה // Lookahead — בדוק שמשהו *קיים* אבל אל תצרף אותו להתאמה /(?=.*\d)/ // Positive lookahead: חייב להיות ספרה אי-שם /(?!.*[אבג])/ // Negative lookahead: לא יכול להיות תו עברי // Groups /(foo|bar)/ // Capturing group — שמור את ההתאמה /(?:foo|bar)/ // Non-capturing group — קבץ ללא שמירה // בדיקת סיסמא: חייבת 8+ תווים, ספרה, אות גדולה, אות קטנה const strongPw = /^(?=.*[A-Z])(?=.*[a-z])(?=.*\d).{8,}$/; strongPw.test('MyPass1!'); // true
אנלוגיהRegExp = מסנן בירידת שוק. "רוצה רק פירות אדומים שמשקלם בין 100-200 גרם" — זו התבנית. lookahead = "רוצה רק דוכנים שנמצאים ליד מאפייה" (בדיקה בלי לעבור לדוכן).
מלכודת נפוצהעם flag g, ל-RegExp יש state פנימי (lastIndex). קריאות חוזרות ל-exec/test על אותו regex יזוזו קדימה. השתמשי ב-new RegExp בכל פעם, או אפסי lastIndex = 0.
בסקריפט שלנוכל 6 כללי הסיסמא מיושמים עם lookahead: /(?=.*[A-Z])/, /(?=.*[a-z])/, /(?=.*\d)/. הבדיקה על תווים אסורים: /[!@#$...]/ — Character class פשוטה. כולם בלי flag g — בדיקת test בלבד.
TYPE טיפוסים typeof, Type Coercion & מלכודות

typeof מחזיר string המתאר את הטיפוס. אבל יש מספר הפתעות ידועות:

typeof 42 // "number" typeof "hello" // "string" typeof true // "boolean" typeof undefined // "undefined" typeof Symbol() // "symbol" typeof BigInt(9) // "bigint" typeof () => {} // "function" // המלכודות הידועות: typeof null // "object" ← BUG היסטורי ב-JS! לא אובייקט! typeof [] // "object" ← מערך הוא object typeof {} // "object" // בדיקות נכונות: Array.isArray([]) // true ← הדרך הנכונה null === null // true ← בדוק null עם === Object.prototype.toString.call([]) // "[object Array]"
אנלוגיהtypeof כמו שאלת "מה זה?" לאדם שבא לדלת. עבור null הוא ישיב "אורח" כשבעצם אין שם אף אחד — זה bug היסטורי שנשמר לצורך backwards compatibility.
Type Coercion מלכודות"5" + 3 = "53" (string concat) אבל "5" - 3 = 2 (number). [] == false הוא true. תמיד השתמשי ב-=== לא ==.
בסקריפט שלנוtypeof password !== 'string' — בדיקת guard clause בראש checkPassword. הטיפוס נבדק לפני כל עיבוד כדי לא לקרוס על null/undefined/number שהועברו בטעות.
ASYNC מנגנון ריצה Event Loop, Call Stack & Microtask Queue

JavaScript היא single-threaded — רק פעולה אחת רצה בו-זמנית. Event Loop מנהל את "התור" של המשימות.

// Call Stack — LIFO: הפונקציה האחרונה שנקראה יוצאת ראשונה function a() { b(); } function b() { console.log('hi'); } a(); // Stack: a → b → console.log → יוצא b → יוצא a // Async: setTimeout → Macro Task Queue setTimeout(() => console.log('timeout'), 0); // Promise.then → Microtask Queue (מועדף על Macro!) Promise.resolve().then(() => console.log('microtask')); console.log('sync'); // סדר הפלט: "sync" → "microtask" → "timeout" // async/await = sugar על גבי Promise async function fetchData() { const res = await fetch('/api/data'); // עצור עד שהPromise נפתר const data = await res.json(); return data; }
אנלוגיהJS כמו קופה אחת בסופר. Event Loop = הקופאית שמסתכלת: Call Stack ריק? אז אקח את הלקוח הבא מהתור. Microtasks (Promise) = לקוח VIP שתמיד נכנס לפני תור ה-setTimeout.
מלכודת נפוצהawait עובד רק בתוך async function. שכחה להוסיף async = SyntaxError. לא להשתמש ב-async/await בתוך forEach — הלולאה לא תחכה. השתמשי ב-for...of עם await.
בסקריפט שלנוהסקריפט הוא synchronous לחלוטין — אין fetch, אין setTimeout, אין Promises. כל הבדיקות רצות ב-Call Stack ישירות, לכן הפלט ב-console מופיע מייד עם הרצה. פשוט ואמין לכלי QA.
VALUES ערכים מיוחדים null, undefined, Falsy & Optional Chaining

JS יש מספר ערכים "ריקים" — חשוב להבין את ההבדל:

// undefined = הוגדר אבל לא הוקצה ערך let x; console.log(x); // undefined // null = ריק בכוונה, הוקצה במפורש let user = null; // "אין משתמש" — כוונה מפורשת // Falsy values: false, 0, "", null, undefined, NaN, 0n // כל שאר הערכים הם Truthy (כולל "0", [], {}!) Boolean("") // false Boolean("0") // true ← מחרוזת לא ריקה! Boolean([]) // true ← מערך ריק! // Optional Chaining ?. — גישה בטוחה לנכסים const city = user?.address?.city; // לא יקרוס אם user=null // Nullish Coalescing ?? — ברירת מחדל רק ל-null/undefined const name = user?.name ?? 'אורח'; // vs || שנותן ברירת מחדל לכל falsy (גם 0, "") const count = userCount ?? 0; // ?? משמר 0 const count2 = userCount || 0; // || יחליף 0 ב-0 — אותו דבר כאן // אבל: userCount = 0: ?? → 0, || → 0. userCount = "": ?? → "", || → 0
אנלוגיהundefined = מגירה שלא נפתחה מעולם. null = מגירה שנפתחה ובכוונה רוקנה. שניהם נראים "ריקים" מבחוץ, אבל הכוונה שונה.
מלכודת נפוצהNaN === NaN הוא false! בדוק עם Number.isNaN(x). גם: null == undefined הוא true (loose equality) אבל null === undefined הוא false.
בסקריפט שלנוguard clause: if (!password || typeof password !== 'string') — !password תופס null, undefined, ומחרוזת ריקה "". זה שימוש חכם ב-falsy לסינון כל הקלטות פסולות בבת אחת.
