🔍 בלוק 2 — const REGEX ביטוי רגולרי

מה זה? ביטוי רגולרי אחד שמייצג את כל 6 הכללים של הסיסמא:

/^(?=.*[A-Za-z])(?=.*[!@#$%*])(?=.*[0-9])(?=.*[a-z])(?!.*[ ~`:^()_+=\-?])(?!.*[\u05D0-\u05EA]).{10,50}$/
חלק ב-Regexסוגמה בודק
^ ... $עוגניםהרגקס חייב לכסות את כל המחרוזת מהתחלה עד הסוף
(?=.*[A-Za-z])Positive Lookaheadחייבת להיות לפחות אות לטינית אחת (גדולה או קטנה)
(?=.*[!@#$%*])Positive Lookaheadחייב תו מיוחד מהרשימה המאושרת בלבד
(?=.*[0-9])Positive Lookaheadחייבת ספרה אחת לפחות
(?=.*[a-z])Positive Lookaheadחייבת אות קטנה אחת לפחות
(?!.*[ ~`:^()_+=\-?])Negative Lookaheadאסור שיהיה תו מהרשימה האסורה
(?!.*[\u05D0-\u05EA])Negative Lookaheadאסורות אותיות עבריות (א–ת בטווח Unicode)
.{10,50}כמותאורך בין 10 ל-50 תווים
מה זה Lookahead? Lookahead ((?=...) / (?!...)) הוא בדיקה קדימה — הרגקס בודק שמשהו קיים (או לא קיים) במחרוזת, מבלי לצרוך תווים. כך אפשר לדרוש כמה תנאים על אותה מחרוזת בבת אחת.
למה \u05D0-\u05EA? אלו קודי ה-Unicode של אותיות א–ת בעברית. הרגקס בודק שאף תו בסיסמא לא נופל בטווח הזה.
🔎 בלוק 3 — function findPasswordInput() פונקציית עזר

מה היא עושה? מחפשת את שדה הסיסמא בדף. מכסה frameworks שונים — כי React, Angular ו-plain HTML יוצרים אלמנטים בצורות שונות.

function findPasswordInput() { const selectors = [ 'input[type="password"]', // שדה סיסמא סטנדרטי 'input[name*="password"]', // שם השדה מכיל "password" 'input[id*="password"]', // ID השדה מכיל "password" 'input[placeholder*="סיסמ"]', // placeholder בעברית ]; for (const sel of selectors) { const el = document.querySelector(sel); if (el) return el; // מחזירה מיד את הראשון שנמצא } return null; // לא נמצא שום שדה }
Selectorמתי מתאים
input[type="password"]plain HTML רגיל
input[name*="password"]שם שדה מכיל "password"
input[id*="password"]ID מכיל "password"
input[placeholder*="סיסמ"]דף בעברית עם placeholder
למה 4 selectors? בדף של משרד הבריאות (Angular), לא תמיד הפקד מוצג עם type="password" בלבד. הכיסוי הרחב מבטיח שהסקריפט עובד גם אחרי עדכוני UI של הדף.
✏️ בלוק 4 — function setInputValue(input, value) פונקציית עזר

הבעיה: ב-Angular/React, כשעושים input.value = 'xxx' ישירות — ה-framework לא מרגיש בשינוי ולא מפעיל validation. הדף נראה כאילו לא הוכנס כלום.

הפתרון — שני שלבים:

function setInputValue(input, value) { // שלב 1: גישה ל-setter המקורי של הדפדפן דרך ה-prototype const setter = Object.getOwnPropertyDescriptor( window.HTMLInputElement.prototype, 'value' )?.set; // setter.call() — מגדיר את הערך דרך המנגנון המקורי setter ? setter.call(input, value) : (input.value = value); // שלב 2: שליחת אירועים ידנית — מפעילה validation ב-Angular/React/Vue ['input', 'change', 'blur'].forEach(e => input.dispatchEvent(new Event(e, { bubbles: true })) ); }
מרכיבמה עושה
Object.getOwnPropertyDescriptorמושך את ה-setter המקורי של input.value מה-prototype של הדפדפן — לפני שה-framework דרס אותו
?.setOptional Chaining — אם אין setter, לא נזרקת שגיאה
setter.call(input, value)מפעיל את ה-setter על ה-input הספציפי
dispatchEvent('input')מפעיל React/Angular לעדכן את ה-state
dispatchEvent('change')מפעיל validators של Angular
dispatchEvent('blur')מפעיל validators שמופעלים רק בעזיבת שדה
bubbles: trueהאירוע "עולה" לאבות ב-DOM — ה-framework שומע אותו
אנלוגיה: כמו שלא ניתן לשנות מחיר ישירות בקופת סופר — צריך לעדכן במחשב הרשמי ולאחר מכן ללחוץ "אשר". setter.call() = עדכון המחשב הרשמי. dispatchEvent = לחיצת "אשר".
💬 בלוק 5 — function getValidationMessage(input) פונקציית עזר

מה היא עושה? קוראת את הודעת השגיאה שה-UI מציג בפועל אחרי הכנסת הסיסמא — לא רק מה הרגקס אומר.

function getValidationMessage(input) { // עולה רמה אחת/שתיים ב-DOM לחפש את הccontainer של השגיאה const parent = input.closest('form') || input.parentElement?.parentElement; // מחפשת אלמנט שגיאה לפי 3 שיטות const el = parent?.querySelector( '[class*="error"],[role="alert"],[aria-live]' ); // מחזירה טקסט, או validationMessage של הדפדפן, או ריק return el?.textContent?.trim() || input.validationMessage || ''; }
שיטהמה מחפשתמתי עובד
[class*="error"]אלמנט עם class שמכיל "error"רוב frameworks
[role="alert"]אלמנט נגישות ARIAAngular Material, Bootstrap
[aria-live]הודעות דינמיות למסכי קריאהדפים עם נגישות גבוהה
input.validationMessageהודעה מובנית של הדפדפןFallback אם לא נמצא כלום
Optional Chaining (?.): כל הגישות כתובות עם ?. — אם האלמנט לא קיים, לא נזרקת שגיאה. el?.textContent?.trim() מחזיר undefined בצורה בטוחה במקום לקרוס.
🚀 בלוק 6 — async function runTests() פונקציה ראשית

הפונקציה הראשית. async כי היא ממתינה לדף בין בדיקות. הזרימה המלאה:

async function runTests() { // שלב 1: חיפוש השדה — עצירה אם לא נמצא const input = findPasswordInput(); if (!input) { console.error('❌ לא נמצא שדה סיסמא!'); return; } const results = []; // שלב 2: לולאה על 20 מקרי הבדיקה for (const tc of TEST_CASES) { // שלב 3: הזנת הסיסמא לשדה setInputValue(input, tc.password); // שלב 4: המתנה 300ms לטעינת ה-validation של הדף await new Promise(r => setTimeout(r, 300)); // שלב 5: בדיקת הרגקס המקומי const regexResult = REGEX.test(tc.password) ? 'PASS' : 'FAIL'; // שלב 6: קריאת הודעת UI מהדף const uiMsg = getValidationMessage(input); // שלב 7: השוואה לצפוי const ok = regexResult === tc.expected; results.push({ ID: tc.id, תיאור: tc.desc, סיסמא: tc.password.substring(0, 14), צפוי: tc.expected, Regex: regexResult, תוצאה: ok ? '✅ עבר' : '❌ נכשל', שגיאה_UI: uiMsg || '-', }); } // שלב 8: ניקוי השדה + הצגת תוצאות setInputValue(input, ''); console.table(results); const passed = results.filter(r => r.תוצאה.includes('✅')).length; console.log(`📊 סיכום: ${passed}/${results.length} עברו`); }
שלבקודהסבר
1findPasswordInput()אם מחזירה null — עצירה עם הודעת שגיאה
2for...of TEST_CASESעובר על כל 20 מקרים לפי הסדר
3setInputValue()מזין את הסיסמא לשדה בצורה שמפעילה Angular
4await setTimeout(300)מחכה 300ms — נותן ל-Angular לבצע validation
5REGEX.test()בדיקת הרגקס המקומי — לא תלוי בדף
6getValidationMessage()קורא מה ה-UI מציג בפועל
7ok = regexResult === expected✅ אם הרגקס מחזיר מה שציפינו, ❌ אחרת
8console.table(results)טבלה מסודרת עם כל התוצאות
למה 300ms ולא פחות? Angular מעבד validation באופן asynchronous — שינוי בשדה לא מוביל מיד להצגת הודעת שגיאה. 300ms נותנים לדף מספיק זמן לעבד את ה-change event ולעדכן את ה-DOM.
