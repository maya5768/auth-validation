בדיקת תווים מיוחדים בשדה סיסמא — מדריך Postman לצוות QA

ה-UI מציג את כל 6 התווים כתקינים — אך השרת מחזיר "משהו השתבש" לחלק מהם. מדריך זה מסביר כיצד לשלוח בקשות HTTP ישירות דרך Postman כדי לקבוע בדיוק אילו תווים השרת מקבל ואילו הוא דוחה.

מערכת: אימות משרד הבריאות תווים נבדקים: ! @ # $ % * כלי: Postman גרסה: 1.0

1 למה פוסטמן?

כאשר ה-UI מציג תגובה כללית כמו "משהו השתבש", קשה לדעת מה קורה בפועל בשרת. דפדפן מבצע בקשות HTTP מאחורי הקלעים — Postman מאפשר לנו לשלוח בדיוק את אותן הבקשות, אך עם שליטה מלאה על כל פרמטר, ולראות בדיוק מה השרת מחזיר.

Postman שולח בקשות HTTP ישירות לשרת, בדיוק כמו הדפדפן — אך עם שליטה מלאה. ניתן לשנות כל פרמטר, לשמור בקשות, לחזור עליהן ולהשוות תוצאות — בלי להיות תלויים ב-UI כלל.

השוואה: DevTools Script מול Postman

קריטריון סקריפט ב-DevTools Console Postman

שקיפות מה קורה תגובה מוצגת ב-Console בלבד; קשה לנתח headers פאנל תגובה מסודר עם Status, Headers, Body — כל דבר בנפרד

2 שלב א — לכידת הבקשה מ-Network

לפני שניתן לשלוח בקשה מ-Postman, עלינו ללכוד את הבקשה שהדפדפן שולח כאשר מבצעים שינוי סיסמא ב-UI. כך נדע בדיוק לאיזה URL לשלוח, אילו headers נדרשים, ואיך נראה ה-body.

  1. פתח את כלי המפתחים: לחץ F12 בדפדפן (Chrome/Edge), ועבור ללשונית Network.
  2. נקה את הלוג: לחץ על כפתור ה-🚫 (Clear) כדי שתראה רק את הבקשות הרלוונטיות לפעולה שלך — לא רעש מרקע של הדף.
  3. בצע את הפעולה ב-UI: מלא סיסמא תקינה (כולל תו מיוחד כלשהו) ולחץ על "שמור" / "אשר". זה גורם לדפדפן לשלוח את הבקשה לשרת.
  4. מצא את הבקשה הרלוונטית: חפש בקשה מסוג POST או PUT בעמודת Method. ה-URL ייראה כמו /api/.../password או /auth/.... לחץ עליה כדי לבחור אותה.
  5. העתק כ-cURL: לחץ קליק ימני על הבקשה הנבחרת → בחר Copy → בחר Copy as cURL (bash). הבקשה המלאה עם כל ה-headers והגוף מועתקת ללוח.

3 שלב ב — ייבוא לפוסטמן

  1. פתח את Postman. אם אין לך אותו, ניתן להוריד בחינם מ-postman.com. אין צורך בחשבון להתחיל.
  2. לחץ על Import (פינה שמאלית עליונה — כפתור כחול). ייפתח חלון ייבוא.
  3. בחר "Raw text" (או "Paste cURL") והדבק את הטקסט שהעתקת מ-DevTools. לחץ Continue ואז Import.
  4. הבקשה נפתחת אוטומטית עם כל ה-headers וה-body ממולאים מהעותק. לא צריך להקליד דבר ידנית.

כך נראה פקודת cURL שנעתקה — זה מה שמדביקים ב-Postman:

curl 'https://api.example.gov.il/auth/password/reset' \
  -X POST \
  -H 'Authorization: Bearer eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9...' \
  -H 'Content-Type: application/json' \
  -H 'X-Request-ID: 4f2a91bc-e3d0-4c1a-a7f9-123456789abc' \
  --data-raw '{"newPassword":"Abc1!defgh","confirmPassword":"Abc1!defgh"}'

הסבר על כל חלק בפקודה:

חלק ב-cURL תפקיד
curl 'https://...' כתובת השרת — לאן הבקשה נשלחת
-X POST מתודה (שיטת HTTP) — POST = שולח נתונים חדשים, PUT = עדכון
-H 'Authorization: Bearer ...' הכי חשוב! הטוקן שמזהה את המשתמש. בלי זה השרת מחזיר 401.
-H 'Content-Type: application/json' אומר לשרת שהגוף הוא JSON
--data-raw '{...}' הגוף — כאן נמצאת הסיסמא שנרצה לשנות
טוקן Authorization: הטוקן תקף לפרק זמן מוגבל (בדרך כלל 15–60 דקות). אם Postman מחזיר שגיאה 401, יש ללכוד בקשה חדשה מה-Network tab ולעדכן את הטוקן.

4 שלב ג — הבנת מבנה הבקשה

כל בקשת HTTP מורכבת משלושה חלקים עיקריים:

🌐
URL
כתובת היעד — לאן הבקשה נשלחת.
דומה לכתובת בית: שם השרת + נתיב ספציפי.
🔑
Headers (כותרות)
מידע על הזהות והפורמט.
Authorization = "המפתח" שמוכיח מי שלח את הבקשה.
📦
Body (גוף)

פירוק ויזואלי של פקודת ה-cURL שלנו עם תיוגים צבעוניים:

5 שלב ד — בדיקת כל תו ידנית

עבור כל אחד מ-6 התווים המיוחדים, נשנה את הסיסמא ב-Body ונשלח את הבקשה. כל בדיקה לוקחת כ-30 שניות. בסך הכל — כ-3 דקות לכל 6 הבדיקות.

  1. לחץ על לשונית Body בחלק העליון של הבקשה ב-Postman. בחר raw ו-JSON מהתפריט הנפתח.
  2. שנה רק את ערך הסיסמא — אל תשנה שום דבר אחר! מצא את "newPassword" ו-"confirmPassword" ושנה את הערך לסיסמא הנבדקת.
  3. לחץ Send (כפתור כתום בפינה הימנית). Postman שולח את הבקשה ומציג את תגובת השרת.
  4. קרא את ה-Status Code בחלק התגובה (ראה סעיף 7 להסבר מפורט). 200 = עבד, 400 = נדחה (validation), 500 = קריסת שרת.
  5. רשום את התוצאה בטבלה למטה לפני שתעבור לתו הבא.
טיפ: אל תלחצי "Send" פעמיים לאותה בקשה — שינוי סיסמא הוא פעולה שמשנה את הסיסמא בפועל. אם תשלחי פעמיים את אותה בקשה, ודאי שה-token עדיין תקף ושהסיסמא הקודמת עדיין מה שנחשבת.

הסיסמאות הנבדקות — 6 תווים:

תו שם התו סיסמא לבדיקה (הדבק ב-Body)

@ At Sign Abc1@defgh
# Hash / Number Sign Abc1#defgh
$ Dollar Sign Abc1$defgh
% Percent Sign Abc1%defgh
* Asterisk Abc1*defgh

כאשר משנים את הסיסמא בלשונית Body של Postman, ה-JSON ייראה כך:

  "confirmPassword": "Abc1!defgh"  // ← ואת זה גם, לאותה ערך
}

6 שלב ה — שימוש ב-Collection Runner (אוטומציה)

Collection Runner מאפשר להריץ קבוצה של בקשות ברצף ולקבל דו"ח מסכם אחד. במקום ללחוץ Send 6 פעמים ולעקוב ידנית — הכל קורה אוטומטית.

  1. שמור את הבקשה ב-Collection: לחץ Save (Ctrl+S) ושמור תחת Collection חדש בשם "בדיקת תווים מיוחדים — סיסמא".
  2. שכפל את הבקשה 6 פעמים: קליק ימני על הבקשה בסייד-בר → Duplicate. עשה זאת 5 פעמים נוספות עד שיש לך 6 עותקים. תן לכל אחד שם: Test ! password, Test @ password, וכן הלאה.
  3. ערוך כל עותק: בכל עותק, שנה את הסיסמא ב-Body לסיסמא הרלוונטית לאותו תו מיוחד.
  4. הפעל את Collection Runner: לחץ על שם ה-Collection בסייד-בר → לחץ על כפתור ▶ (Run). ייפתח Collection Runner.
  5. לחץ Run: כל 6 הבקשות ירוצו ברצף. כל תוצאה תוצג מיד.

7 קריאת התגובה — Response

לאחר לחיצה על Send, חלק התגובה (Response) מוצג בחצי התחתון של Postman. הנה איך לקרוא אותו:

Status Code משמעות פעולה נדרשת
200 OK הבקשה הצליחה — הסיסמא שונתה בהצלחה תעדי כ-PASS
400 Bad Request השרת דחה את הבקשה (validation error) — הבקשה לא תקינה לפי הגדרות תעדי כ-FAIL, בדקי את ה-Response Body להסבר
401 Unauthorized הטוקן פג תוקף או שגוי לכודי בקשה חדשה עם טוקן מעודכן
403 Forbidden אין הרשאות לבצע את הפעולה בדקי שהמשתמש מחובר ויש לו הרשאת שינוי סיסמא
500 Internal Server Error קריסה בצד השרת — השרת לא ידע לטפל בבקשה תעדי כ-BUG! זה פגם — שגיאת validation בשרת, לא ב-UI

9 מה עושים עם התוצאות

פרשנות לפי Status Code:

Status שהתקבל משמעות פעולה נדרשת
200 OK התו עובד — השרת מקבל את הסיסמא ומשנה אותה תעדי תוצאה חיובית. אין צורך בפגם.
400 Bad Request השרת דוחה את הסיסמא עם הסבר — ה-UI צריך להציג הודעת שגיאה ברורה בדקי את ה-Response Body לפרטי השגיאה. אם ה-UI לא מציג את הסיבה לדחייה — פתחי בעיה קלה (UI חסר).
500 Internal Server Error באג! השרת קרס — לא טיפל בתו זה בצורה בטוחה. ה-UI מציג את התו כתקין אך השרת לא מוכן לקבלו. פתחי דוח פגם מיידי. השרת צריך לדחות את התו עם 400 ולא לקרוס ב-500. הצרף לדוח: תו, סיסמא, Status, ו-Response Body מלא.

תבנית דוח פגם — להעתיק ל-Azure DevOps / Jira:

כותרת: תו [X] גורם לשגיאת שרת 500 בשינוי סיסמא

תיאור:
בעת שינוי סיסמא המכילה את התו [X] (לדוגמה: # — Hash), ה-UI מציג את הסיסמא כתקינה (פס הסיסמא ירוק, כפתור "שמור" פעיל), אך השרת מחזיר שגיאה 500 Internal Server Error.

שלבים לשחזור:
1. היכנס למערכת ועבור לדף שינוי סיסמא
2. הזן סיסמא המכילה את התו [X]: Abc1#defgh
3. לחץ שמור / אישור
4. (אופציונלי) שלח בקשה ישירה דרך Postman לנקודת הקצה:
   POST https://api.example.gov.il/auth/password/reset
   Body: {"newPassword":"Abc1#defgh","confirmPassword":"Abc1#defgh"}

תוצאה צפויה:
השרת מחזיר תגובה מסודרת — אם התו לא מותר, יש להחזיר Status 400 עם הודעת שגיאה ברורה. ה-UI צריך להציג את ההודעה למשתמש.

תוצאה בפועל:
ה-UI מציג: "משהו השתבש. יש לנסות שוב."
Postman מחזיר: Status 500 — Internal Server Error
Response Body: {"success":false,"error":"Internal Server Error","message":"משהו השתבש. יש לנסות שוב."}

חומרה: High — ה-UI מטעה את המשתמש לחשוב שהסיסמא תקינה
סביבה: Production / Staging
מערכת הפעלה / דפדפן: Windows 10 / Chrome 120
מזהה בקשה (requestId): 4f2a91bc-e3d0-4c1a (מה-Response Body)
רמז לשורש הבעיה: תווים כמו % ו-# הם תווים מיוחדים בפרוטוקולי URL ו-HTTP. % מתחיל רצף Percent-Encoding, ו-# מסמן Fragment בURL. ייתכן שהשרת לא מבצע Escape נכון לתווים אלו לפני שמשתמש בהם. המפתח צריך לוודא שהסיסמא מטופלת כ-plain string ולא מפורשת כחלק מ-URL.
